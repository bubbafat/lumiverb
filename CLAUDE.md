# Claude Code — Lumiverb repository guide

A navigation index. The goal is to make a fresh conversation productive
without burning tokens on exploration. Prefer reading this file before
`ls` / `grep` / `find`. Details that drift (logic, schemas, thresholds)
are **intentionally omitted** — read the code. File paths and
responsibility mappings change less often; that's what lives here.

---

## Tech stack

- **Server**: Python 3.12, FastAPI, SQLModel, PostgreSQL 16, Quickwit (BM25).
  Schema migrations use **Alembic** — control plane + per-tenant trees in
  `migrations/`, configs in `alembic-control.ini` / `alembic-tenant.ini`,
  driver in `scripts/migrate.sh`. A separate **custom data-backfill
  runner** lives in `src/server/upgrade/` (registered steps in
  `upgrade/steps/`, exposed via `/v1/tenant/upgrade`) — that's for
  long-running online data migrations, not DDL.
- **CLI / workers**: Python 3.12. Entry point `lumiverb = src.client.cli:main`
  from `pyproject.toml`. Uses `httpx` to talk to the server.
- **Web UI**: React 18 + TypeScript + Vite + Tailwind 3 + TanStack Query v5
  + TanStack Virtual v3 + React Router v6. Dev server proxies `/v1/*` to
  the API.
- **Native macOS / iOS**: Swift 5.9, SwiftUI, CoreML, Apple Vision. Swift
  package `LumiverbKit` is shared by both platforms; macOS and iOS app
  targets live in an XcodeGen-managed Xcode project.
- **Auth**: Hybrid JWT (web login) + API keys (CLI/automation). Both use
  `Authorization: Bearer <token>`. Roles: `admin`, `editor`, `viewer`.

---

## Runtime topology

The FastAPI server is the only persistent service. The Python CLI, the
web UI, and the native macOS/iOS apps are all clients that talk to it
over `/v1/`. Two of those clients additionally do **local enrichment**
and POST results back: the Python CLI runs InsightFace / faster-whisper
in subprocesses (`src/client/`), and the macOS app runs CLIP / ArcFace /
OCR via CoreML + Vision (`clients/lumiverb-app/Sources/macOS/Enrich/`).
The iOS app is browse-only. Search is offloaded to a Quickwit sidecar
(BM25), with a Postgres fallback path.

## Running things

Standard incantations (`uvicorn src.server.api.main:app --reload`,
`lumiverb <cmd>`, `npm run dev` / `build` in `src/ui/web`, `swift build` /
`swift test` in `LumiverbKit`) work as you'd expect. Only the
non-obvious ones are listed here:

**macOS app build**  `cd clients/lumiverb-app && xcodegen generate && xcodebuild -project Lumiverb.xcodeproj -scheme Lumiverb-macOS -configuration Debug build CODE_SIGNING_ALLOWED=NO`
**Convert ArcFace**  `cd scripts/convert-models && .venv/bin/python convert_arcface.py` (requires its own venv — see script docstring)

**uv only** — `pip` is not available. Use `uv add <pkg>`, `uv run <cmd>`,
`uv sync` (feedback memory).

## Tests

**Python** — `tests/` with pytest markers (`pyproject.toml`):
- `fast` — no DB, no AI, mocks/tmp_path only. Run: `uv run pytest -m fast`
- `slow` — testcontainers Postgres. Run: `uv run pytest -m slow`
- `ai` — requires real inference (Moondream Station running)
- Full suite: `uv run pytest` (runs fast + slow; ai is opt-in)

**Swift (LumiverbKit)** — XCTest in the package test target. Fixtures in
`Sources/LumiverbKit/Tests/LumiverbKitTests/Fixtures/` (face_single.jpg,
face_group.jpg, face_crowd.jpg). There is **no macOS-target test bundle**
— anything that needs to be unit-tested must live in LumiverbKit, not in
`Sources/macOS/`.

**Web UI** — no component render tests today (see
`docs/project_ui_testing_gap.md` in the user's memory).

## API conventions (do not relitigate)

- All routes under `/v1/`.
- Every route requires `Authorization: Bearer <token>`.
- Tenant context derived from the token. **Never a URL param.**
- Pagination is **cursor-based**, never offset. `next_cursor` in responses,
  `after` in query params.
- Error envelope: `{"error": {"code": "...", "message": "...", "details": {...}}}`.
- File uploads are multipart form.
- No webhooks, no source-file serving, no rate limiting in v1.
- Admin-only endpoints use `Depends(require_admin)` (global admin key) or
  `Depends(require_tenant_admin)` (tenant role).
- Routes are matched by FastAPI in definition order — static paths MUST
  precede parameterized siblings.

## Gotchas worth knowing before editing

- **`active_assets` view** must be dropped + recreated when adding columns
  to the `assets` table. See feedback memory.
- **ONNX Runtime leaks memory** in long-running Python worker batches
  (~35 MB/inference). Batch work is isolated into subprocesses for exactly
  this reason. Don't "simplify" that into a single process.
- **`systemd` units are not regenerated** by `scripts/update-vps.sh` after
  entry-point changes — must be fixed manually.
- **`ThreadPoolExecutor` submit pattern**: never submit all futures at once.
  Cap at `concurrency * 2` and use `_drain()`. See feedback memory.
- **Prod access**: never SSH or run commands on the prod VPS; the user
  handles deployments. The server is auto-updated after push.
- **The `src/` legacy directories** (`api/`, `cli/`, `core/`, etc.) are
  empty `__pycache__` shells from pre-ADR-014. Real code is under
  `src/server/`, `src/client/`, `src/shared/`.
