# Claude Code ‚Äî Lumiverb repository guide

A navigation index. The goal is to make a fresh conversation productive
without burning tokens on exploration. Prefer reading this file before
`ls` / `grep` / `find`. Details that drift (logic, schemas, thresholds)
are **intentionally omitted** ‚Äî read the code. File paths and
responsibility mappings change less often; that's what lives here.

---

## Tech stack

- **Server**: Python 3.12, FastAPI, SQLModel, PostgreSQL 16, Quickwit (BM25).
  Alembic for DDL; custom runner in `src/server/upgrade/` for long-running
  online data backfills. See the DDL / backfill rows below.
- **CLI / workers**: Python 3.12. Entry point `lumiverb = src.client.cli:main`
  from `pyproject.toml`. Uses `httpx` to talk to the server.
- **Web UI**: React 18 + TypeScript + Vite + Tailwind 3 + TanStack Query v5
  + TanStack Virtual v3 + React Router v6. Dev server proxies `/v1/*` to
  the API.
- **Native macOS / iOS**: Swift 5.9, SwiftUI, CoreML, Apple Vision. Swift
  package `LumiverbKit` is shared by both platforms; macOS and iOS app
  targets live in an XcodeGen-managed Xcode project.
- **Auth**: Hybrid JWT (web login) + API keys (CLI/automation). Both use
  `Authorization: Bearer <token>`. Roles: `admin`, `editor`, `viewer`.
- **Topology**: FastAPI is the only persistent service. CLI + macOS run
  local enrichment and POST results back. iOS is browse-only. Quickwit
  sidecar with a Postgres fallback path.

## Running things

Standard incantations work as expected (`uvicorn src.server.api.main:app
--reload`, `lumiverb <cmd>`, `npm run dev` in `src/ui/web`, `swift test`
in `LumiverbKit`). Non-obvious ones:

- **uv only** ‚Äî `pip` is not available. Use `uv add` / `uv run` / `uv sync`.
- **macOS app build** ‚Äî `cd clients/lumiverb-app && xcodegen generate && xcodebuild -project Lumiverb.xcodeproj -scheme Lumiverb-macOS -configuration Debug build CODE_SIGNING_ALLOWED=NO`
- **Convert ArcFace** ‚Äî `cd scripts/convert-models && .venv/bin/python convert_arcface.py` (its own venv; see script docstring)

## Tests

**Python** ‚Äî `tests/` with pytest markers (`pyproject.toml`):
- `fast` ‚Äî no DB, no AI, mocks/tmp_path only. Run: `uv run pytest -m fast`
- `slow` ‚Äî testcontainers Postgres. Run: `uv run pytest -m slow`
- `ai` ‚Äî requires real inference (Moondream Station running)
- Full suite: `uv run pytest` (runs fast + slow; ai is opt-in)

**Swift (LumiverbKit)** ‚Äî XCTest in the package test target. Fixtures in
`Sources/LumiverbKit/Tests/LumiverbKitTests/Fixtures/` (face_single.jpg,
face_group.jpg, face_crowd.jpg). There is **no macOS-target test bundle**
‚Äî anything that needs to be unit-tested must live in LumiverbKit, not in
`Sources/macOS/`.

**Web UI** ‚Äî no component render tests today (see
`docs/project_ui_testing_gap.md` in the user's memory).

## API conventions (do not relitigate)

- All routes under `/v1/`.
- Every route requires `Authorization: Bearer <token>`.
- Tenant context derived from the token. **Never a URL param.**
- Pagination is **cursor-based**, never offset. `next_cursor` in responses,
  `after` in query params.
- Error envelope: `{"error": {"code": "...", "message": "...", "details": {...}}}`.
- File uploads are multipart form.
- No webhooks, no source-file serving, no rate limiting in v1.
- Admin-only endpoints use `Depends(require_admin)` (global admin key) or
  `Depends(require_tenant_admin)` (tenant role).
- Routes are matched by FastAPI in definition order ‚Äî static paths MUST
  precede parameterized siblings.

## Gotchas worth knowing before editing

- **`active_assets` view** must be dropped + recreated when adding columns
  to the `assets` table. See feedback memory.
- **ONNX Runtime leaks memory** in long-running Python worker batches
  (~35 MB/inference). Batch work is isolated into subprocesses for exactly
  this reason. Don't "simplify" that into a single process.
- **`systemd` units are not regenerated** by `scripts/update-vps.sh` after
  entry-point changes ‚Äî must be fixed manually.
- **`ThreadPoolExecutor` submit pattern**: never submit all futures at once.
  Cap at `concurrency * 2` and use `_drain()`. See feedback memory.
- **Prod access**: never SSH or run commands on the prod VPS; the user
  handles deployments. The server is auto-updated after push.
